What changed

01

Strict GPU Plonk path

Compression and aggregation Plonk proofs run with `cpuFallback=false`. The GPU host is now useful end-to-end instead of only accelerating isolated MSMs.

02

BLS12-377 PI Vortex MiMC

CUDA builds the MiMC Merkle trees for SIS and no-SIS commitments. This removes the old Merkleization wall in the public-input wizard.

03

Fused large SIS commitments

The 1880-row and 812-row PI commitments use fused GPU SIS+MiMC. Smaller SIS commitments stay on CPU because the upload/setup overhead is not yet worth it; the faster two-column kernel still leaves 288 rows around break-even and 108 rows slower than CPU.

04

Quotient cache and map cleanup

Quotient roots are cached in coefficient form, coset domains are reused, and hot `sync.Map` traffic is gone. The first large quotient now sits around 29 seconds.

05

Two-column BLS SIS tile

The fused BLS SIS leaf kernel now handles two encoded columns per CUDA block. On the representative aggregation request, wall time moved from the final 4:44 average range to a 4:35 single-proof spot check.

Why this is safe to keep

The optimization does not modify circuit definitions, gnark `Define()` functions, public input layouts, verification keys, or wizard constraint registration. It changes how existing witness-side commitments and quotient evaluations are computed.

CPU equivalence tests GPU SIS hashes, no-SIS MiMC hashes, Merkle leaves, internal nodes, and roots are compared against the CPU implementation.
Proof sanity checks The final batch produced 3 compression checks and 9 aggregation sub-proof checks across BLS12-377, BW6-761, and BN254.
Conservative fallback Unsupported smartvector shapes, unsupported SIS parameters, allocation failure, or CUDA errors fall back to the existing CPU path.
Protocol boundary unchanged No circuit structs, `Define()` methods, verification keys, transcript order, or wizard constraint registrations changed. The GPU path only changes deterministic witness-side BLS12-377 hashing and quotient computation.

Still on the table

The next big step is a BLS12-377 device-resident Vortex path, analogous to the existing KoalaBear GPU Vortex architecture but implemented for this PI commitment scheme's BLS12-377 MiMC/ring-SIS stack. The current retained path accelerates commitments after the encoded rows are already on the host. A resident BLS Vortex state would keep the encoded rows or a column-major snapshot on device, compute UAlpha after the `Alpha` coin without another full transfer, and download only selected columns and selected SIS digests after the `Q` coin. That is the design likely to move the wizard from a 2-minute critical path toward the original 10x target.

The remaining large hotspot is BLS12-377 coset reevaluation inside quotient computation: 2173 non-constant roots over a 262144 domain take about 17.7 seconds. A batched GPU coset FFT API that avoids one cgo call and one host/device round trip per root is the next high-leverage step; one-root-at-a-time GPU prototypes were intentionally discarded, including a BLS12-377 version that worsened this sub-step to 20.6 seconds. A batched transfer/API variant also matched the CPU transform but worsened the same sub-step to 27.5 seconds, so it was removed. The SIS path can also improve if the key/twiddle buffers become resident and small commitments stop paying fixed upload costs. The latest CUDA timing confirms that the two big fused SIS commitments are dominated by the BLS SIS leaf kernel itself. The retained two-column tile reduced that leaf kernel from 14.1s to 8.4s for the 1880-row commitment, and from 12.7s to 7.2s for the 812-row commitment.

Bottom line: the GPU path is no longer a science project. It is a faster and cheaper production direction with clear correctness guardrails and obvious next optimizations.
