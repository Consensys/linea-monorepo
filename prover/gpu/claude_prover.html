Linea zkEVM Prover Deep Dive & GPU Acceleration Strategy for 40B-Cell Traces

February 2026 — Confidential Technical Report

1. High-Level Overview

The Linea prover transforms EVM execution traces into succinct zero-knowledge proofs verifiable on Ethereum L1. The system operates over the KoalaBear field (p = 231 − 227 + 1 ≈ 2.13 × 109), a 31-bit prime optimized for 32-bit arithmetic. The pipeline can be decomposed into five macro-phases:

┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐ │ 1. TRACE LOAD │ ──▶ │ 2. WIZARD IOP │ ──▶ │ 3. QUOTIENT + │ ──▶ │ 4. VORTEX PCS │ ──▶ │ 5. OUTER PROOF │ │ Parse .lt file │ │ Compile arith. │ │ CONSTRAINT │ │ Commit + Open │ │ gnark PLONK │ │ Expand columns │ │ → polynomial │ │ EVALUATION │ │ RS + SIS + MT │ │ BLS12→BW6→BN254│ │ Pad to 2^k │ │ IOP │ │ │ │ │ │ │ └─────────────────┘ └─────────────────┘ └─────────────────┘ └─────────────────┘ └─────────────────┘ ~5% ~5% ~40% ~35% ~15%

For a 40 billion cell trace (≈ thousands of columns × millions of rows), the dominant costs are in phases 3 and 4: the quotient polynomial computation and the Vortex polynomial commitment scheme. Together, these account for roughly 75% of total proving time.

2. Glossary of Key Terms

Term Definition
KoalaBear The base field: 𝔽p where p = 231 − 227 + 1. Each element is 4 bytes. Supports NTT with max order root 224.
𝔽p4 Degree-4 extension of KoalaBear: 𝔽p²[u]/(u²−3), then 𝔽p⁴[v]/(v²−u). Each element = 16 bytes. Used for Fiat-Shamir challenges and polynomial commitments.
Column A vector of field elements representing one register across all rows of a trace module. Stored as a SmartVector.
Module A logical grouping of columns representing an EVM component (e.g., hub, mmu, keccak). Each module has its own height limit.
Wizard The custom polynomial-IOP framework. An CompiledIOP accumulates columns, constraints, and random coins through multiple "rounds" of interaction.
Global Constraint A polynomial identity that must hold over an entire domain H of size n: C(x) ≡ 0 for all x ∈ H.
Quotient Polynomial Q(x) = C(x) / (xn − 1). Proving C vanishes on H is equivalent to showing Q is a polynomial. Split into ratio many shares.
Ratio For a constraint of degree d over domain of size n: ratio = ⌈(d−n+1)/n⌉ rounded up to next power of 2. Determines how many coset evaluations are needed.
Vortex A polynomial commitment scheme based on Reed-Solomon codes + Ring-SIS hashing + Merkle trees. Polynomials are laid out as a matrix (rows × columns), RS-encoded, then column-hashed into a Merkle tree.
Ring-SIS A lattice-based hash: H(m) = Σi Ai · mi mod (Xd+1). Parameters: degree=512, logTwoBound=16. Collision-resistant and algebraically structured.
Reed-Solomon Systematic encoding: IFFT → zero-pad → FFT on larger domain. Rate ρ (typically 2) means the encoded vector is ρ× the original size.
Self-Recursion The Vortex verifier is itself expressed as a circuit in the Wizard framework, enabling proof composition without a separate recursive step.
Arcane The master compilation pipeline that reduces a high-level Wizard protocol into a single-point polynomial IOP suitable for Vortex.
SmartVector Memory-optimized polynomial representation: Regular (dense), Constant (single value), PaddedCircularWindow (sparse with padding), Rotated (lazy shift).

4. Trace & Witness Generation

4.1 What Is a Trace?

A trace is the complete record of an EVM execution, organized as a matrix: each module is a sub-matrix, and each column within a module represents a register (e.g., program counter, stack pointer, memory address) across all execution steps (rows).

Module "hub" (2^21 rows × ~200 columns): Row PC OPCODE STACK_PTR GAS_LEFT ... ───── ──── ─────── ────────── ───────── ──── 0 0 PUSH1 0 30000000 1 2 PUSH1 1 29999997 2 4 ADD 2 29999994 ... ... ... ... ... 2^21-1 (zero-padded) Module "keccak" (2^18 rows × ~150 columns): Row ROUND A[0][0] A[0][1] ... ───── ────── ──────── ──────── ──── ...

4.2 Field Elements

Every cell in the trace is a KoalaBear field element — a 31-bit integer stored in Montgomery form in 4 bytes:

// maths/field/element.go
type Element = koalabear.Element  // 4 bytes, Montgomery form

// Field parameters:
//   p = 2^31 - 2^27 + 1 = 2,130,706,433
//   Multiplicative generator: 3
//   Max NTT size: 2^24 (16M elements)
//   Montgomery constant R = 2^32 mod p
Why KoalaBear? A 31-bit prime enables fast 32-bit arithmetic on both CPU and GPU. Two elements fit in a 64-bit register, and Montgomery reduction needs only a single multiply + shift. The multiplicative group supports NTT up to size 224, large enough for most column sizes.

4.3 Extension Field

Fiat-Shamir challenges and polynomial commitments operate over a degree-4 extension:

// maths/field/fext/doc.go
// 𝔽r²[u] = 𝔽r / (u² − 3)      → quadratic extension
// 𝔽r⁴[v] = 𝔽r² / (v² − u)     → quartic extension

type Element = extensions.E4  // 4 × KoalaBear = 16 bytes
// Contains: B0.A0, B0.A1, B1.A0, B1.A1

6. Quotient Polynomial Computation

This is the single most expensive step in the prover. File: protocol/compiler/globalcs/quotient.go.

6.1 The Mathematical Problem

Given a polynomial constraint C(X) that must vanish on a domain H = {ω0, ω1, ..., ωn-1}:

# Python pseudocode
def compute_quotient(constraint_expr, witness_columns, domain_H):
    """
    If C(x) = 0 for all x in H, then (x^n - 1) | C(x).
    We compute Q(x) = C(x) / (x^n - 1) and split it into shares.
    """
    n = len(domain_H)

    # Determine degree: if C has degree d, quotient has degree d - n
    # ratio = ceil((d - n + 1) / n), rounded up to power of 2
    ratio = next_power_of_2(ceil((degree(constraint_expr) - n + 1) / n))

    # For each coset i in [0, ratio):
    quotient_shares = []
    for i in range(ratio):
        # 1. Compute shift for this coset
        shift = generator_full_group * omega_coset^i
        coset_domain = [shift * omega^j for j in range(n)]

        # 2. Re-evaluate each witness column on the coset
        #    This is the expensive part: IFFT + FFT per column
        for col in unique_root_columns:
            coeffs = IFFT(col.values, domain_H)        # O(n log n)
            col_on_coset = FFT(coeffs, coset_domain)    # O(n log n)

        # 3. Apply shifts (rotations are free — just pointer offset)
        for shifted_col in shifted_columns:
            shifted_col_on_coset = rotate(root_col_on_coset, offset)

        # 4. Evaluate constraint expression (DAG evaluation)
        constraint_values = expr_board.evaluate(col_values_on_coset)

        # 5. Divide by vanishing polynomial: (x^n - 1)^{-1}
        annulator_inv = precomputed_annulator_inverse[i]
        quotient_share = constraint_values * annulator_inv

        quotient_shares.append(quotient_share)

    return quotient_shares

7. Vortex Polynomial Commitment Scheme

7.1 Architecture

Vortex commits to polynomials by organizing them as a matrix, Reed-Solomon encoding each row, hashing columns with Ring-SIS, and building a Merkle tree:

Step 1: Arrange polynomials as matrix rows Poly₁: [a₁ a₂ a₃ ... aₙ] ← n = NbColumns (e.g., 1024) Poly₂: [b₁ b₂ b₃ ... bₙ] ... Polyₘ: [m₁ m₂ m₃ ... mₙ] ← m = NbRows (e.g., 256) Step 2: RS-encode each row (ρ=2 → doubles width) Enc₁: [a₁ â₁ a₂ â₂ ... aₙ âₙ] ← 2n elements Enc₂: [b₁ b̂₁ b₂ b̂₂ ... bₙ b̂ₙ] ... Encₘ: [m₁ m̂₁ m₂ m̂₂ ... mₙ m̂ₙ] Step 3: Hash each COLUMN with Ring-SIS + Poseidon2 Col₁: SIS([a₁,b₁,...,m₁]) → [h₁₁...h₁ₛ] → Poseidon2 → leaf₁ Col₂: SIS([â₁,b̂₁,...,m̂₁]) → [h₂₁...h₂ₛ] → Poseidon2 → leaf₂ ... Col₂ₙ: → leaf₂ₙ Step 4: Build Merkle tree ROOT / \ H(0,1) H(2,3) / \ / \ leaf₁ leaf₂ leaf₃ leaf₄ ...

7.2 Reed-Solomon Encoding — The FFT Cost

// crypto/reedsolomon/reedsolomon.go — RS encoding per row
func (p *RsParams) RsEncodeBase(v SmartVector) SmartVector {
    expandedCoeffs := make([]field.Element, p.NbEncodedColumns()) // 2n
    v.WriteInSlice(expandedCoeffs[:v.Len()])

    // Fast path for rate=2:
    inputCoeffs := expandedCoeffs[:p.NbColumns()]
    p.Domains[0].FFTInverse(inputCoeffs, fft.DIF, fft.WithNbTasks(1))
    inputCoeffs.Mul(inputCoeffs, p.CosetTableBitReverse)
    p.Domains[0].FFT(inputCoeffs, fft.DIT, fft.WithNbTasks(1))

    // Interleave original and encoded values
    for j := p.NbColumns()-1; j >= 0; j-- {
        expandedCoeffs[2*j+1] = expandedCoeffs[j]  // encoded
        expandedCoeffs[2*j] = v.Get(j)              // original
    }
    return NewRegular(expandedCoeffs)
}
Critical observation: fft.WithNbTasks(1) means each individual RS encoding runs single-threaded. The parallelism happens at the row level (multiple rows encoded in parallel via parallel.Execute). This is the right strategy when you have thousands of rows, but wasteful when a single large domain FFT could benefit from parallelism.

7.3 Ring-SIS Transversal Hash

After RS encoding, each column (across all rows) is hashed with Ring-SIS:

// crypto/ringsis/ringsis.go — TransversalHash (simplified)
func (s *Key) TransversalHash(v []SmartVector, res []field.Element) {
    parallel.Execute(nbCols, func(start, end int) {
        // Windowed transposition for cache efficiency
        const windowSize = 32
        const padding = 8  // avoid cache set aliasing

        transposedArena := arena.NewVectorArena(paddedNbRows * windowSize)

        for col := start; col < end; col += windowSize {
            // Transpose 32-column window
            for row := 0; row < nbRows; row++ {
                for j := 0; j < windowSize; j++ {
                    transposed[j][row] = v[row].Get(col + j)
                }
            }
            // Hash each column in window
            for j := 0; j < windowSize; j++ {
                s.SisGnarkCrypto.Hash(transposed[j], result)
            }
        }
    })
}

8. Self-Recursion & Outer Proof

8.1 The Curve Tower

KoalaBear (31-bit) Inner IOP + Vortex PCS │ ▼ BLS12-377 (253-bit scalar) gnark PLONK circuit verifies Wizard proof │ ~128M constraints ▼ BW6-761 (761-bit scalar) Aggregation: verifies multiple BLS12-377 proofs │ BW6 scalar field = BLS12-377 base field (native!) ▼ BN254 (254-bit scalar) Final proof for Ethereum on-chain verification

8.2 Self-Recursion

Instead of a separate recursive circuit, the Vortex verifier is embedded back into the Wizard framework itself. The self-recursion compiler (protocol/compiler/selfrecursion/) performs these steps:

  1. PlonkInWizard context: The non-Vortex part of the proof is verified as PLONK constraints within the Wizard
  2. Vortex context translation: Commitments, coins, and Merkle roots are mirrored in the outer circuit
  3. Three prover actions: ConsistencyCheck, AssignVortexUAlpha (linear combination), AssignVortexOpenedCols

8.3 Gnark PLONK Proving

The final outer proof uses gnark's PLONK implementation:

12. GPU FFT Strategy

12.1 KoalaBear NTT on GPU

KoalaBear's 31-bit modulus is perfect for GPU NTT:

# Python pseudocode for GPU KoalaBear NTT kernel

MODULUS = 2130706433  # 2^31 - 2^27 + 1

def butterfly_kernel(a, b, twiddle):
    """Single NTT butterfly — maps to 2 GPU multiply-adds."""
    # All ops in uint32, Montgomery form
    t = montgomery_mul(b, twiddle)  # 1 multiply + 1 reduce
    return (a + t) % MODULUS, (a - t + MODULUS) % MODULUS

def batch_ntt_gpu(data, n_polynomials, poly_size, twiddles):
    """
    Launch kernel with grid = (n_polynomials, poly_size / 2).
    Each thread handles one butterfly across log2(poly_size) stages.
    """
    for stage in range(log2(poly_size)):
        # Early stages: butterflies within warp (shared memory)
        # Late stages: butterflies across blocks (global memory)
        butterfly_stage_kernel<<>>(
            data, twiddles[stage], stage, poly_size
        )

12.3 Handling NTT Size > 224

The KoalaBear field supports NTT only up to size 224. For larger domains, use mixed-radix or Bluestein's algorithm. However, in practice, trace modules are capped at 221–223, well within limits.

13. GPU Hashing Strategy

13.1 Ring-SIS on GPU

Ring-SIS hashing is fundamentally FFT-based polynomial multiplication:

# Per-column Ring-SIS hash (currently on CPU)
def ring_sis_hash_gpu(column_values, sis_key):
    """
    For each polynomial A[i] in SIS key:
        k = input_limbs[i*degree : (i+1)*degree]
        k_fft = NTT(k)
        a_fft = NTT(A[i])  # precomputed
        result_fft += k_fft * a_fft   # element-wise
    result = INTT(result_fft)
    """
    # GPU parallelism: batch all columns together
    # 2048 columns × 512-degree NTTs = batch NTT of size 2048×512
    return batch_ring_sis(all_columns, precomputed_key_ffts)

13.2 Poseidon2 on GPU

Poseidon2 is already optimized with SIMD (16-column batches via CompressPoseidon2x16). GPU can extend this to thousands of parallel hashes:

  • Each Poseidon2 permutation: ~8 MDS matrix multiplications + S-box layers
  • All 2n leaf hashes can run in parallel: one thread per leaf
  • The MDS matrix is small (width 8 for KoalaBear octuplets)

14.3 Pipeline Architecture

# Proposed GPU pipeline — overlaps compute with transfer

async def gpu_vortex_commit(witness_columns, gpu_pool):
    # Phase 1: Stream witness to GPUs (overlapped with previous work)
    futures = []
    for gpu_id, chunk in partition(witness_columns, 8):
        f = gpu_pool[gpu_id].upload_async(chunk)
        futures.append(f)

    # Phase 2: RS encode on each GPU (embarrassingly parallel)
    encoded = []
    for gpu_id in range(8):
        e = gpu_pool[gpu_id].batch_rs_encode_async()  # NTTs on GPU
        encoded.append(e)

    # Phase 3: Ring-SIS hash on each GPU
    hashes = []
    for gpu_id in range(8):
        h = gpu_pool[gpu_id].batch_sis_hash_async()
        hashes.append(h)

    # Phase 4: Gather hashes to host, Poseidon2 + Merkle on host/GPU
    all_hashes = gather(hashes)
    leaves = poseidon2_batch_gpu(all_hashes, gpu_pool[0])
    root = merkle_tree_host(leaves)

    return root, encoded

16. Proposed Algorithmic Improvements

16.1 Fuse IFFT+FFT in Quotient Computation

Currently, each quotient coset evaluation does IFFT (to get coefficients) then FFT (to evaluate on coset). These can be fused into a single "coset shift" operation:

# Current: 2 FFTs per column per coset
coeffs = IFFT(values_on_H)       # O(n log n)
values_on_coset = FFT(coeffs, coset)  # O(n log n)

# Proposed: 1 FFT + element-wise multiply
# If coset = shift * H, then evaluating on coset is:
#   f(shift·ωʲ) for all j
# Which equals: IFFT(values) → multiply by shift_powers → FFT
# But we can precompute the "coset transfer matrix" and apply in one pass

coeffs = IFFT(values_on_H)
coeffs *= coset_twiddles[:]  # element-wise, O(n)
values_on_coset = FFT(coeffs)  # same domain, no shift

# This saves creating new Domain objects and their twiddle tables

16.2 Cache Root Column FFTs Across Cosets

Currently, the arena is reset between cosets, discarding previously computed coefficient representations. If we cache the IFFT result (coefficient form) of each root column, subsequent coset evaluations only need a single FFT:

# Current flow (quotient.go)
for coset in range(maxRatio):
    arenaExt.Reset(0)  ← discards previous IFFT results!
    for col in roots:
        coeffs = IFFT(witness)   ← redundant for coset > 0!
        vals = FFT_on_coset(coeffs)

# Proposed: cache coefficient form
coeff_cache = {}
for col in roots:
    coeff_cache[col] = IFFT(witness)  ← once per column

for coset in range(maxRatio):
    for col in roots:
        vals = FFT_on_coset(coeff_cache[col])  ← only 1 FFT/col/coset
Impact: This eliminates ~50% of FFT calls in the quotient computation (all the IFFTs after the first coset). For 200 columns × 4 cosets: saves 600 FFTs → ~30s on CPU, sub-second on GPU.

16.3 Batch SIS Hash with Precomputed NTT Keys

The SIS key polynomials A[i] can be precomputed in NTT form once during setup. The current code computes FFT(A[i]) for every hash call. Precomputing saves one NTT per key polynomial per hash invocation.

16.4 Reduce Extension Field Usage

Many operations currently use 𝔽p4 (16 bytes per element) where 𝔽p (4 bytes) would suffice. The extension is needed only for Fiat-Shamir challenges. Profile which columns are always in the base field and add fast-path branching:

// Already partially done in quotient.go:
if smartvectors.IsBase(witness) {
    // 4× faster path using base field FFT
    res := arena.Get[field.Element](arenaExt, ctx.DomainSize)
    domain0.FFTInverse(res, fft.DIF, fft.WithNbTasks(2))
    domain.FFT(res, fft.DIT, fft.OnCoset(), fft.WithNbTasks(2))
} else {
    // Extension field path (4× slower)
    res := arena.Get[fext.Element](arenaExt, ctx.DomainSize)
    domain0.FFTInverseExt(res, fft.DIF, fft.WithNbTasks(2))
    domain.FFTExt(res, fft.DIT, fft.OnCoset(), fft.WithNbTasks(2))
}

16.5 Increase FFT Parallelism

Currently fft.WithNbTasks(2) is used throughout. On a 96-core machine, each individual FFT could use 4-8 threads for the initial butterfly stages, improving latency when fewer columns are being processed in parallel.

16.6 Streaming Merkle Tree Construction

Instead of computing all leaf hashes first and then building the tree, stream: as soon as two adjacent leaves are ready, compute their parent. This enables overlapping hash computation with tree building and reduces peak memory.

16.7 Unified GPU Memory Manager

Replace per-operation arena allocators with a unified GPU memory manager that:

  • Pre-allocates all GPU memory at startup based on column count and domain sizes
  • Uses pinned host memory for zero-copy transfers
  • Implements a pool allocator for temporary FFT buffers
  • Supports async memcpy overlapped with compute
