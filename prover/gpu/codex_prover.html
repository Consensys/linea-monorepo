Linea prover deep dive + GPU acceleration architecture report

This report is the requested part 1 + part 2: exact proving flow from execution request to final proof, then a concrete multi-GPU integration strategy focused on the heavy kernels in crypto/vortex, protocol/compiler/globalcs/quotient.go, and the final gnark outer proof path.

Repo: linea-monorepo/prover Date: February 18, 2026 Assumed scale: 40,000,000,000 cells

0) Scope & assumptions

  • This is static code-path analysis (no heavy compile/run on this machine, per your constraint).
  • Numbers are engineering estimates using code structure and element sizes; they are not profiler measurements.
  • For 40B cells, this report assumes those are base-field witness cells before RS encoding.
  • AWS instance specs are taken from AWS EC2 G7e docs as of Feb 18, 2026.

1) Key terms

Term Definition in this codebase
CompiledIOP Static protocol object: columns, queries, coins, prover/verifier actions by round (protocol/wizard/compiled.go).
ProverRuntime Mutable runtime state during proving: assigned columns, query params, coins, and transient state map (protocol/wizard/prover.go).
Arcane Compiler pipeline reducing generic constraints into polynomial checks + quotient/opening machinery (protocol/compiler/arcane.go).
Vortex Polynomial commitment system used after Arcane, with RS encoding + Merkle proofs + optional SIS hashing.

Outer proof Final gnark proof that verifies the recursed wizard proof and checks functional execution inputs (circuits/execution).

2) End-to-end flow (request → final proof)

2.1 Request ingestion and job dispatch

Entry path cmd/prover/cmd/prove.go decides job type from input filename and routes execution jobs to backend/execution.Prove.

# Python-like pseudocode
req = json.load(input_file)
if "getZkProof" in input_name:
    resp = execution.Prove(cfg, req, large)
write_json(output_file, resp)

2.2 Functional output + witness creation

CraftProverOutput parses blocks/logs, computes execution-data multi-commitment, and assembles public-functional input context. Then NewWitness links trace file + signatures + block hashes + execution-data commitment witness.

// backend/execution/craft.go
execDataMultiCommitment := public_input.ComputeExecutionDataMultiCommitment(execDataBuf.Bytes())

2.3 Full-mode proving split: inner then outer

In full mode (backend/execution/prove.go):

  • Load/compile full zkEVM IOP.
  • Generate inner proof: fullZkEvm.ProveInner.
  • Sanity verify inner proof.
  • Generate final outer gnark proof via circuits/execution.MakeProof.
// backend/execution/prove.go (full mode)
proof := fullZkEvm.ProveInner(w.ZkEVM)
if err := fullZkEvm.VerifyInner(proof); err != nil { ... }
return execution.MakeProof(traces, setup, fullZkEvm.RecursionCompiledIOP, proof, *w.FuncInp, w.ZkEVM.ExecData)

2.4 zkEVM module assignment order

Main assignment function is ZkEvm.GetMainProverStep in zkevm/zkevm.go. Critical ordering: arithmetization first, then dependent modules.

// zkevm/zkevm.go
z.Arithmetization.Assign(run, input.ExecTracesFPath)
z.Ecdsa.Assign(run, ...)
z.StateManager.Assign(run, ...)

z.PublicInput.Assign(run, ...)

2.5 Arcane + Vortex compilation suites

zkevm/full.go chains multiple suites (poseidon2, Arcane, Vortex, self-recursion, cleanup), then a second post-recursion suite. This creates a staged protocol where the heavy commitment/opening logic is repeated on progressively compressed instances.

2.6 Inner proof recursion boundary

ZkEvm.ProveInner runs initial prover only up to VortexQueryRound(initial)+1, verifies up to same round, extracts recursion witness, then proves the recursion-compiled IOP.

// zkevm/zkevm.go
stoppingRound := recursion.VortexQueryRound(z.InitialCompiledIOP) + 1
initialRun := wizard.RunProverUntilRound(..., stoppingRound, ...)
recWitness := recursion.ExtractWitness(initialRun)
finalProof := wizard.Prove(z.RecursionCompiledIOP, ...)

2.7 Outer gnark circuit semantics

circuits/execution/circuit.go verifies the recursed wizard transcript in-circuit, checks extracted public inputs against functional execution inputs, checks execution-data commitment, and enforces final public-input hash equality.

// circuits/execution/circuit.go
c.WizardVerifier.Verify(api)
checkPublicInputs(api, &c.WizardVerifier, c.FuncInputs, c.ExecDataBytes)
api.AssertIsEqual(c.PublicInput, c.FuncInputs.Sum(api))

3) Runtime state machine (what state exists at each phase)

S0
Raw request JSON loaded, config loaded.
S1
Functional response fields populated (BlocksData, hashes, state roots, execution-data checksum).
S2
zkevm.Witness created: trace filepath + tx signatures + block hashes + commitment opening data.
S3
ProverRuntime created: precomputed columns loaded, FS seeded with FiatShamirSetup.
S4
Round-0 assignments: full arithmetization and all module witnesses inserted into run.Columns.
S5
Compiler-generated round actions execute: quotient shares, Vortex commitments, query params, random coins.
S6
Vortex stores heavy transient artifacts in run.State: encoded matrices, Merkle trees, SIS/noSIS hashes.
S7
Vortex opening phase: linear combination Ualpha, selected columns, packed Merkle proofs assigned.
S8
Partial proof extracted at recursion boundary; final FS state captured.
S9
Recursion witness materialized (proof + committed matrices + trees + hashes + public inputs).
S10
Recursion compiled IOP fully proven → final inner proof.
S11
Outer gnark assignment built, wizard verifier circuit assigned, final proof generated and serialized.

Fiat-Shamir transition logic

On each round transition, prover appends proof-visible messages and query params to FS transcript, increments round, samples new coins (goNextRound() in protocol/wizard/prover.go).

// protocol/wizard/prover.go
msgsToFS := run.Spec.Columns.AllKeysInProverTranscript(run.currRound)
for each msg in msgsToFS: FS.Update(msg)
for each params in queryParamsAtRound: params.UpdateFS(FS)
run.currRound++
for each coin in coinsAtNewRound: run.Coins[name] = Sample(FS, seed)

4) Hotspots: exact heavy zones

4.1 Global constraints quotient (protocol/compiler/globalcs/quotient.go)

  • Builds quotient shares by coset evaluation, with repeated FFT/IFFT over involved root columns.
  • Computes shifted columns via SoftRotate over reevaluated roots.
  • Uses a big arena: arena.NewVectorArena[fext.Element](DomainSize * len(AllInvolvedRoots)).
// protocol/compiler/globalcs/quotient.go
arenaExt := arena.NewVectorArena[fext.Element](ctx.DomainSize * len(ctx.AllInvolvedRoots))
...
domain0.FFTInverse(res, fft.DIF, fft.WithNbTasks(2))
domain.FFT(res, fft.DIT, fft.OnCoset(), fft.WithNbTasks(2))

4.2 Vortex commitment/opening path (protocol/compiler/vortex/prover.go)

  • Per commitment round: RS encode + column hashing + Merkle root assignment.
  • Stores heavy state by round in run.State (encodedMatrix, tree).
  • Later opening round reads all matrices + trees, selects random columns, packs Merkle proofs. 
// protocol/compiler/vortex/prover.go
run.State.InsertNew(ctx.VortexProverStateName(round), committedMatrix)
run.State.InsertNew(ctx.MerkleTreeName(round), tree)
...
entryList := run.GetRandomCoinIntegerVec(ctx.Items.Q.Name)
vortex_koalabear.SelectColumnsAndMerkleProofs(&proof, entryList, committedMatrices, trees)
// protocol/compiler/vortex/prover.go
committedMatrix := run.State.MustGet(ctx.VortexProverStateName(round)).(vortex_koalabear.EncodedMatrix)
run.State.Del(ctx.VortexProverStateName(round)) // frees heavy object once consumed

4.3 Crypto kernels inside Vortex

  • RsEncodeBase (FFT-based RS encoding) in crypto/reedsolomon/reedsolomon.go.
  • TransversalHash (matrix transpose + SIS hash per encoded column) in crypto/ringsis/ringsis.go.
  • Poseidon2 leaf hashing (SIS and no-SIS modes) in crypto/vortex/vortex_koalabear/commtiment.go.
  • Linear combination over encoded rows in crypto/vortex/prover_common.go.

4.4 Outer proof (gnark)

Final proving call is circuits.ProveCheck in circuits/execution/circuit.go. It includes the in-circuit wizard verifier, which itself replays all verifier-round logic and FS coin generation in-circuit.

5) 40B-cell memory/CPU model

5.1 Size assumptions from code

5.2 Matrix model used here

Assume: N = 40,000,000,000 base cells (pre-RS committed witness volume) rho = 2 (common Vortex blow-up) NumCols = 2^22 = 4,194,304 (typical large target column size) Rows ~= N / NumCols ~= 9,537 EncodedCols = NumCols * rho = 8,388,608

5.2.1 Gap vs current devnet-full limits

Metric Value
Configured trace modules 71

Sum of large limits (limit_large) 28,146,437 cells

The 40B scenario is far beyond static full/large limits from config/config-devnet-full.toml; it requires a scalable/limitless-style execution strategy plus memory-aware scheduling.

5.3 Memory footprint (order-of-magnitude)

Component Formula Estimate
Raw base witness N * 4 ~149 GiB
RS-encoded matrices N * rho * 4 ~298 GiB
Extra transpose/scratch pass ~N * rho * 4 up to ~298 GiB
Merkle leaves + internal nodes (per commitment batch) ~2 * EncodedCols * 32 ~0.5 GiB
GlobalCS arena (example) DomainSize * Roots * 16, with 2^22, 256 roots ~16 GiB
Implication: At 40B cells, naive CPU-only residency of raw + encoded + scratch alone is already ~745 GiB before other runtime objects. This is why state-lifetime control and tiled processing are mandatory.

5.4 CPU work model (ballpark)

Kernel Work estimate Why it hurts
RS encode ~8.8e11 FFT butterflies (with above dimensions) Dominant arithmetic + memory traffic
SIS/noSIS transversal hashing touches ~8.0e10 encoded cells transpose + hash; cache/memory bound
Linear combination ~8.0e10 extension MAC-style ops full scan over encoded rows
GlobalCS quotient FFTs for 256 roots, 4 cosets: ~8.6e9 point-level FFT passes repeated domain transforms
Expected CPU behavior: near-flat 100% utilization across available cores during these kernels. Wall-clock is then bounded by memory bandwidth + FFT implementation throughput + allocation churn.

6.3 Data-layout and execution strategy

  • Use row-major batches for RS encode, then column tiles for transversal hash.
  • Pin host memory for large movement boundaries; overlap transfer + compute using streams.
  • Avoid whole-matrix residency on one device; tile by row chunks and/or encoded-column chunks.
  • Keep FS and transcript sequencing on CPU exactly as-is; GPU kernels must be pure compute subroutines.

6.4 Pseudocode for multi-stage Vortex GPU commit

7.2 Fit analysis for 40B cells

  • Raw base witness (~149 GiB) + encoded (~298 GiB) cannot be safely duplicated in VRAM with scratch.
  • Therefore: streamed/tiled execution is required even with 4×96GB.
  • Host RAM (1024 GiB) is enough for large staging windows if state lifetimes are controlled.

7.3 Recommended partitioning

Across rounds
  • Assign independent commitment rounds to devices if round sizes are balanced.
  • Good for reducing cross-device synchronization.
Within one big round
  • Split by row chunks (e.g. 512–1024 rows/chunk at 2^22 cols).
  • Then split encoded columns per device for hash/tree stage.

7.4 Concrete chunk sizing example

Chunk rows Raw chunk (GiB) Encoded chunk (GiB, rho=2) Comment
512 ~8 ~16 Conservative, easy overlap
1024 ~16 ~32 Good throughput/overhead trade-off
1536 ~24 ~48 Aggressive but still practical with 96GB
If each GPU keeps encoded + scratch + leaf/tree buffers, practical occupancy per device is usually 50–75 GB, not full 96 GB.

8) Fundamental simplifications to make acceleration easier

8.1 Replace "store everything in run.State" with streaming lifetimes

  • Today Vortex stores encoded matrices and trees by round until opening round.
  • Introduce a staged artifact manager: allow spill-to-disk/mmap or immediate compaction of tree layers.
  • This directly lowers peak RAM and eases GPU pipeline overlap.

8.2 GlobalCS quotient: cache coefficient form across cosets

  • Current path repeatedly does IFFT + FFT per root/coset combination.
  • Cache root coefficients once per root, then only coset FFT per coset.
  • Cuts repeated transforms and lowers both CPU and GPU work.

8.3 Unify commitment modes under one batch API

  • Currently no-SIS and SIS branches diverge in multiple places.
  • Create one batch commitment API with mode flag; move branching to backend.
  • Improves kernel fusion opportunities and simplifies device scheduling.

8.4 Introduce accelerator capability matrix

  • Not all kernels should be GPU-first at day one.
  • Build feature-gated backend map: rs_encode, sis_hash, poseidon2, linear_comb, quotient_fft.
  • Lets you deploy partial acceleration without protocol-level risk.

8.5 Explicit deterministic transcript checks in CI

  • Any GPU path must preserve exact proof bytes / FS state transitions.
  • Add cross-backend equivalence tests (CPU vs GPU) per round and final proof.

9) Delivery roadmap (pragmatic)

Phase Scope Expected outcome
Phase A GPU RS encode + linear combination only Immediate speedup with low transcript risk
Phase B GPU SIS/noSIS hashing + Merkle build Major Vortex speedup and CPU offload
Phase C GPU GlobalCS quotient FFT path Fix second major bottleneck
Phase D Outer proof acceleration (if gnark backend supports your curve stack) Reduce tail latency
Recommended first implementation target: Vortex-only acceleration path (RS + hash + Merkle + linear-comb) behind a runtime flag. This captures most ROI before touching recursion/outer-circuit logic.
